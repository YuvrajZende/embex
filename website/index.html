Embex

Local-first memory layer for AI agents — semantic search, CRAG Q&A, and persistent memory over your codebase. Zero cloud. Zero cost.

What It Does

  • Embeds your codebase into a local ChromaDB vector database in one command
  • Answers natural-language questions about code using CRAG (retrieve → filter → LLM answer with file citations)
  • Snapshots every change automatically, so you can restore deleted or modified files
  • Stores agent memories semantically, surviving across sessions
  • Exposes everything as an MCP server — plug Cursor, Claude Desktop, or Cline directly into your codebase

Key Features

FeatureDescription
Semantic SearchFind code by meaning with embex query
CRAG Q&ALLM answers grounded in your code with embex ask
Agent MemoryPersistent semantic memory with embex memory
File HistoryAuto-snapshots, diff, and restore with embex log / embex restore
MCP Server8 tools via Model Context Protocol with embex serve-mcp
Fully LocalNo Docker, no cloud, no API keys for core functionality

Requirements

  • Python 3.10+
  • No API keys for embeddings or search (runs fully offline)
  • z.ai API key only needed for LLM features (ask, explain --llm)

Installation

Get Embex running on your machine in under a minute. No Docker. No cloud accounts. No API keys for core functionality.

Prerequisites

  • Python 3.10 or newer
  • pip (comes with Python)
  • Git

Install from Source

bash 
git clone https://github.com/YuvrajZende/embex
cd embex && pip install -e .
That's it. No Docker, no cloud accounts, no API keys needed for core functionality.

LLM Setup (Optional)

The ask and explain --llm commands use an LLM. Create a global env file:

dotenv 
# ~/.embex/.env
ZAI_API_KEY=your_key_here

Default model: glm-4.7-flash (free tier). Available models: glm-5, glm-4.7, glm-4.7-flash, glm-4.5-flash.

Quick Start

From zero to your first semantic search in 60 seconds.

Step 1: Initialize

Navigate to your project and run embex init. This scans every code file, embeds them, and stores everything locally in .embex/.

bash
$ cd /path/to/your-project
$ embex init

Step 2: Search

Run a semantic query — find code by meaning, not keywords:

bash
$ embex query "authentication flow"

Step 3: Ask Questions

Use the CRAG pipeline to get grounded answers with citations:

bash
$ embex ask "how does JWT refresh work"

Step 4: Watch for Changes

Keep embeddings fresh as you code:

bash
$ embex watch
  Watching for changes... (Ctrl+C to stop)

Step 5: Explain a File

bash
$ embex explain src/auth/token.py         # static analysis
$ embex explain src/auth/token.py --llm  # LLM-powered summary

CLI Reference

Complete reference for all embex commands, flags, and output examples.

embex init

Initializes Embex in a project directory. Creates .embex/, scans every code file, embeds them (skipping unchanged files via checksum cache), and stores everything locally.

Re-running embex init is fully idempotent — it skips unchanged files, processes new or modified files, and automatically removes deleted files.

bash
$ embex init                        # current directory
$ embex init C:\Projects\my-app     # specific path

embex watch

Watches for file changes. On every save: snapshots the version, re-embeds the file.

bash
$ embex watch

embex query

Semantic search — finds the most relevant code chunks for a natural-language query.

bash
$ embex query "how does user authentication work"
$ embex query "database connection" --top-k 10
$ embex query "error handling" --folder src/utils
FlagDefaultDescription
--top-k5Number of chunks to return
--folder—Scope search to a subfolder

embex ask

CRAG pipeline — retrieves relevant chunks, filters by similarity, sends to LLM for a grounded, cited answer.

bash
$ embex ask "how does authentication work"
$ embex ask "where are database queries made" --folder src/db
$ embex ask "explain the chunking strategy" --top-k 10
FlagDefaultDescription
--top-k8Chunks to retrieve
--folder—Scope to a subfolder
--threshold0.30Min similarity score (0–1)
--no-sources—Hide the source panel

embex explain

Analyzes a file — static analysis by default, LLM summary with --llm.

bash
$ embex explain src/auth/login.py           # static
$ embex explain src/auth/login.py --llm     # LLM-powered

embex memory

Persistent, semantically searchable memory for AI agents. Stored in .embex/memory.db.

bash
# Store a decision
$ embex memory add "We use HS256 JWT tokens for API auth"
$ embex memory add "Payment service uses Stripe" --tags payments,stripe

# Retrieve by meaning
$ embex memory recall "authentication approach"
$ embex memory recall "payment processing" --top-k 3

# Manage
$ embex memory list
$ embex memory forget a1b2c3d4
$ embex memory clear --yes

embex log

Shows the full version history of a file.

bash
$ embex log src/auth/login.py

embex diff

Side-by-side diff between two versions of a file.

bash
$ embex diff src/auth/login.py              # last two versions
$ embex diff src/auth/login.py --v1 1 --v2 3

embex restore

Restores a file to any previous version.

bash
$ embex restore src/auth/login.py --version 1

embex status

Shows current database contents — what's in ChromaDB and SQLite.

embex search

Exact text or regex search (different from query which is semantic).

bash
$ embex search "def login"
$ embex search "TODO:.*" --regex

embex stats

Shows aggregate project health metrics — file count, disk usage, and churn hotspots.

embex similar

Finds files semantically similar to a given file — useful for spotting duplicate logic.

bash
$ embex similar src/auth/login_v2.py

embex ignore

Adds patterns to the exclusion list in embex.json.

bash
$ embex ignore "*.log"
$ embex ignore "temp_data" --dir

Python SDK

Give any AI agent full codebase awareness in 10 lines of Python.

Basic Usage

python
from embex import EmbexSDK

sdk = EmbexSDK("/path/to/your-project")

Semantic Search

python
results = sdk.search("authentication logic", top_k=5)
for r in results:
    print(r["file"], r["score"])

Q&A with Citations

python
answer = sdk.ask("How does JWT refresh work?")
print(answer["answer"])
print(answer["sources"])

Agent Memory

Persistent memory that survives across sessions:

python
sdk.remember("User prefers async/await over callbacks", tags=["code-style"])
memories = sdk.recall("coding style preferences")

File History

python
history = sdk.get_file_history("src/auth/token.py")
old_version = sdk.get_file_version("src/auth/token.py", version_id=history[0]["id"])

MCP Integration

Connect Cursor, Claude Desktop, Windsurf, Cline, or any MCP client to your codebase instantly.

Starting the Server

bash
$ cd /path/to/your-project
$ embex serve-mcp

Example Agent Session

session
Agent: What files handle payment processing?
  → calls search_codebase("payment processing")
  → src/payments/stripe.py (0.891), src/payments/webhook.py (0.843)

Agent: How does the webhook validation work?
  → calls ask_codebase("webhook validation")
  → "In src/payments/webhook.py, validate_signature() computes..."

Agent: Remember: we switched from PayPal to Stripe in Feb 2026
  → calls remember("switched from PayPal to Stripe", tags=["payments"])
  → Memory stored.

Notes

  • The MCP server is stateless per request — it reads from .embex/ on each call
  • Multiple agents can share the same MCP server instance and memory store
  • Tag memories with --agent <name> to namespace them per agent

Configuration

Embex is configured via .embex/embex.json in your project root. Run embex init to generate it with sensible defaults.

Full Schema

json
{
  "version": "1.0",
  "project_name": "my-app",
  "embedding": {
    "provider": "local",
    "model": "all-MiniLM-L6-v2"
  },
  "llm": {
    "provider": "zai",
    "model": "glm-4.7-flash"
  },
  "rag": { "top_k": 8, "relevance_threshold": 0.30, "max_chunk_chars": 1200 },
  "chunking": { "strategy": "ast", "chunk_size": 200, "overlap": 20 },
  "history": { "enabled": true, "max_versions_per_file": 50 }
}

Embedding Settings

FieldDefaultDescription
provider"local""local" — free, runs offline. "openai" — paid, requires API key.
model"all-MiniLM-L6-v2"Any sentence-transformers model for local provider.

LLM Settings

FieldDefaultDescription
provider"zai""zai" — z.ai GLM models. "ollama" — fully local, no API key.
model"glm-4.7-flash"Model name. For z.ai: glm-5, glm-4.7, glm-4.7-flash.

z.ai Model Options

ModelSpeedQuality
glm-4.7-flashFastGood — default, best balance
glm-4.7MediumBetter — full flagship
glm-5SlowerBest — latest, most capable

RAG Settings

FieldDefaultDescription
top_k8Number of code chunks retrieved initially
relevance_threshold0.30Min similarity score (0–1) for a chunk to be relevant
max_chunk_chars1200Max characters per chunk sent to LLM

Chunking Settings

FieldDefaultDescription
strategy"ast""ast" — splits on function/class boundaries. "fixed" — splits by line count.
chunk_size200Lines per chunk (only for "fixed" strategy)
overlap20Overlapping lines between chunks

API Key Setup

Embex looks for API keys in this order:

  1. Project-local .env — in the project root
  2. Global fallback — ~/.embex/.env
dotenv
# ~/.embex/.env — create once, works for all projects

Architecture

How Embex processes, stores, and queries your codebase.

Pipeline Overview

text
Your codebase
      |
      v
 embex init / watch
      |
  Scanner + Chunker
  SHA-256 checksums skip unchanged, prune deleted
      |
  Embedder (sentence-transformers)
  all-MiniLM-L6-v2
      |
  ChromaDB (vectors)  +  SQLite (history + memory)
      |
  Query / Ask / MCP
  retrieve → filter → LLM → cited answer

System Architecture

text
+----------------------------------------------+
|           CLI  /  SDK  /  MCP                |
+----------------------------------------------+
                    |
+----------------------------------------------+
| Scanner → Smart Chunker (AST) → Embedder     |
|               |            → VectorStore      |
|               +----------→ HistoryStore       |
|                                               |
| CRAG Pipeline: VectorStore → filter → LLM    |
| MemoryStore: isolated from code vectors       |
+----------------------------------------------+
     |            |            |          |
 ChromaDB     SQLite       SQLite     MCP Server
 (vectors)   (history)    (memory)

Key Design Decisions

  • AST-based chunking: Splits code on function and class boundaries, not arbitrary line counts. This preserves semantic meaning in each chunk.
  • SHA-256 checksums: Re-running init is idempotent — unchanged files are skipped, deleted files are pruned.
  • Isolated memory store: Agent memories are stored in a separate SQLite database, not mixed with code vectors.
  • CRAG pipeline: Corrective Retrieval-Augmented Generation — retrieves chunks, filters by relevance threshold, then generates grounded answers.

Adding Custom Extensions

You can add more extensions by editing the include_extensions field in your .embex/embex.json configuration file.

json
{
  "watch": {
    "include_extensions": [".py", ".js", ".ts", ".vue", ".svelte"]
  }
}
